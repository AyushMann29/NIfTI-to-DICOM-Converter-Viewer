# Medical Imaging Viewer: NIfTI to DICOM Web Viewer

A full-stack web application designed for medical professionals and researchers to upload, convert, and visualize 3D medical scans. This tool seamlessly converts NIfTI files (`.nii.gz`) into a web-viewable format, complete with segmentation masks, using a modern, responsive, and efficient architecture.

This project was developed as part of an internship application to demonstrate proficiency in full-stack development, system architecture, and medical imaging data handling.

## ‚ú® Core Features

* **NIfTI File Upload:** Drag-and-drop or browse to upload 3D medical images and their corresponding segmentation masks.
* **Asynchronous Backend Processing:** Uploads are processed in the background, providing an immediate UI response and preventing browser timeouts on large files. The frontend polls for status updates.
* **NIfTI-to-DICOM Conversion:** Converts volumetric NIfTI images into a series of DICOM slices on the fly.
* **Segmentation Mask Handling:** Processes NIfTI segmentation masks and converts them into a DICOM-SEG format, linking them to the source image series.
* **Interactive Web Viewer:**
    * **Slice-by-Slice Navigation:** Scroll through 2D slices of the 3D volume using a slider or next/previous buttons.
    * **Pre-Generated Previews:** All slices are pre-converted to PNGs for blazing-fast loading and smooth navigation.
    * **Segmentation Overlay Control:** Toggle the visibility of different anatomical structures defined in the mask file.

## üèóÔ∏è System Architecture

The application is designed to be highly responsive and scalable by decoupling the file upload from the heavy processing tasks.

1.  **Upload:** The user uploads an image and mask file. The React frontend sends these to the FastAPI backend.
2.  **Immediate Response:** The backend immediately saves the files, adds the conversion task to a **background queue**, and returns a `202 Accepted` response with a unique `study_id`.
3.  **Frontend Polling:** The React app receives the `study_id` and begins polling a `/status` endpoint every few seconds. It displays a "Processing..." overlay to the user.
4.  **Background Conversion:** Meanwhile, the background task on the server performs all the heavy lifting:
    * Converts the NIfTI image into a series of DICOM slices.
    * Generates a DICOM-SEG file from the mask.
    * **Pre-renders all slices into PNG images** for fast frontend loading.
    * Creates a `metadata.json` file upon completion.
5.  **Completion:** Once the frontend's polling request sees that `metadata.json` exists, it knows the processing is complete.
6.  **Data Fetch:** The frontend makes a final request to an `/instances` endpoint to get the URLs for all the pre-rendered PNGs and segmentation data.
7.  **Viewing:** The data is passed to the DicomViewer component, allowing for instant, lag-free browsing of the image slices.

This asynchronous architecture ensures a smooth user experience, even with very large medical scan files.

## üöÄ API Endpoints

The backend exposes the following RESTful API endpoints:

| Method | Endpoint                             | Description                                                                                                                             |
| :----- | :----------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------- |
| `POST` | `/upload`                            | Accepts multipart form data with `image_file`, `mask_file`, and `radiography_type`. Kicks off background processing and returns a `study_id`. |
| `GET`  | `/studies`                           | Returns a list of all previously processed studies.                                                                                     |
| `GET`  | `/studies/{study_id}/status`         | Checks if the background processing for a given study is complete. Returns `{ "status": "processing" }` or `{ "status": "complete" }`.      |
| `GET`  | `/studies/{study_id}/instances`      | Returns the final data for a completed study, including URLs to the static PNG previews and the DICOM-SEG file.                           |
| `GET`  | `/uploads/{study_id}/previews/{slice_name}.png` | **(Static)** Serves the pre-generated PNG slice previews. This is handled by FastAPI's `StaticFiles`.                                     |

## ‚öôÔ∏è Setup and Installation

To run this project locally, you will need **Python 3.8+** and **Node.js v16+**.

### Backend Setup

1.  **Clone the repository:**
    ```bash
    git clone https://github.com/AyushMann29/NIfTI-to-DICOM-Converter-Viewer
    cd NIfTI-to-DICOM-Converter-Viewer/backend
    ```

2.  **Create a virtual environment and activate it:**
    ```bash
    # For Windows
    python -m venv venv
    .\venv\Scripts\activate

    # For macOS/Linux
    python3 -m venv venv
    source venv/bin/activate
    ```

3.  **Install Python dependencies:**
    *(Create a `requirements.txt` file in your `backend` folder with the content below first)*
    ```bash
    pip install -r requirements.txt
    ```

4.  **Run the backend server:**
    ```bash
    uvicorn main:app --reload
    ```
    The server will be running at `http://127.0.0.1:8000`.

**`backend/requirements.txt`:**
fastapi
uvicorn[standard]
python-multipart
pydicom
highdicom
nibabel
numpy
Pillow
### Frontend Setup

1.  **Navigate to the frontend directory:**
    ```bash
    cd frontend
    ```

2.  **Install Node.js dependencies:**
    ```bash
    npm install
    ```

3.  **Run the React development server:**
    ```bash
    npm run dev
    ```
    The application will open automatically in your browser at `http://localhost:3000`.
